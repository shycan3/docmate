# 중요 심사 질문과 답변 모음

작성일: 2026-05-20  
용도: 발표 질의응답 대비용 핵심 답변집

## 1. 그냥 ChatGPT에 PDF 넣고 분석하면 되는 것 아닌가요?

맞습니다. 범용 생성형 AI도 PDF를 요약할 수 있습니다. 다만 DocMate는 범용 요약 도구가 아니라 장학금·청년정책 신청 준비에 특화된 업무 흐름입니다.

사용자가 매번 프롬프트를 쓰는 대신 프로필을 선택하고 공고를 넣으면, 신청 가능 여부, 위험 조건, 제출 서류, 원문 근거, 체크리스트, 히스토리 비교가 항상 같은 형식으로 나옵니다. 현재는 생성형 AI를 쓰지 않는 규칙 기반 데모라 비정형 문서 해석에는 한계가 있지만, 재현성과 설명 가능성, 로컬 개인정보 처리, 반복 사용 흐름에서 차별점이 있습니다.

짧은 답변:

> 생성형 AI는 문서를 읽어주는 도구이고, DocMate는 신청 준비 과정을 표준화하는 서비스입니다.

## 2. 모든 PDF를 분석할 수 있나요?

아닙니다. 현재 버전은 모든 PDF를 범용으로 요약하는 서비스가 아닙니다. 텍스트 추출이 가능하고, `신청 기간`, `지원 대상`, `제출 서류`, `신청 방법`처럼 일정한 항목 구조가 있는 장학금·청년정책 공고에 적합합니다.

스캔 이미지 PDF, OCR이 필요한 문서, 표와 각주가 복잡하게 섞인 문서, 자유서술형 보고서는 현재 한계가 큽니다. 그래서 발표에서는 "모든 PDF 요약"이 아니라 "구조화된 정책·장학금 공고를 신청 준비 흐름으로 바꾸는 도구"라고 설명하는 것이 정확합니다.

## 3. 생성형 AI를 쓰지 않는데 요약은 어떻게 하나요?

DocMate의 결과는 생성형 요약이 아니라 규칙 기반 정보 추출과 재구성입니다.

1. PDF에서 텍스트를 추출합니다.
2. 공고 안의 `신청 기간`, `지원 대상`, `제출 서류` 같은 라벨을 찾습니다.
3. 해당 라벨 아래 내용을 표준 항목에 넣습니다.
4. `만 18세 이상`, `부산 거주`, `소득 2분위 이하` 같은 조건을 정규식으로 찾습니다.
5. 사용자 프로필과 비교해 기본 조건 충족 여부를 판정합니다.
6. 위험 조건, 원문 근거, 체크리스트를 생성합니다.

즉, 사람이 새 요약문을 쓰듯 생성하는 방식이 아니라 공고의 구조를 찾아 신청 준비 데이터로 바꾸는 방식입니다.

## 4. 규칙 기반 분석이면 정확도가 낮지 않나요?

규칙 기반 분석은 비정형 문서 해석에는 약하지만, 반복되는 형식의 공고에서는 재현성과 설명 가능성이 좋습니다. 같은 입력이면 같은 결과가 나오고, 어떤 원문을 근거로 추출했는지 보여줄 수 있습니다.

현재 버전은 발표 데모 기준이며, 실제 서비스 정확도를 주장하려면 실공고 fixture를 모아 항목 추출 정확도, 위험 조건 탐지, 자격 판정 결과를 측정해야 합니다. 그래서 향후 로드맵에는 20~50개 이상의 실제 공고 기반 평가셋 구축을 포함했습니다.

## 5. 신청 가능이라고 나왔는데 실제로는 불가능하면 어떻게 하나요?

DocMate는 최종 심사 권한을 가진 서비스가 아니라 신청 준비 보조 도구입니다. 그래서 애매한 조건은 가능하다고 단정하지 않고 `추가 확인 필요`로 남기는 보수적 판정을 사용합니다.

발표에서는 `신청 가능`을 "기본 조건 기준 신청 가능"이라고 설명하는 것이 안전합니다. 최종 제출 전에는 반드시 원문 공고와 기관 안내를 확인해야 하며, DocMate는 그 확인 과정을 빠르게 돕는 도구입니다.

## 6. 원문 근거는 어떻게 보여주나요?

추출 항목별로 라벨이 포함된 줄을 먼저 찾고, 없으면 추출값과 일치하는 문장을 찾습니다. 위험 조건은 키워드가 등장한 문장을 근거로 연결합니다.

예를 들어 `제출 서류`가 추출되면, 결과 화면에서 해당 항목의 원문 문장을 펼쳐 볼 수 있습니다. 이 방식은 자동 분석 결과를 맹신하지 않게 하고, 사용자가 직접 원문과 대조할 수 있게 합니다.

## 7. 최신 정책 정보는 어떻게 보장하나요?

현재 로컬 데모는 사용자가 업로드한 공고 또는 내장 샘플을 분석합니다. 따라서 최신성은 사용자가 넣은 문서 기준입니다.

실제 서비스로 확장하려면 공고 수집 파이프라인, 원문 링크 검증, 갱신일 표시, 기관별 공고 데이터 관리가 필요합니다. 현재 버전은 "최신 공고를 자동 수집하는 서비스"가 아니라 "입력된 공고를 신청 준비용으로 구조화하는 서비스"입니다.

## 8. 개인정보는 어떻게 처리하나요?

현재 데모는 로컬에서 실행되고 외부 AI API를 호출하지 않습니다. 원본 업로드 파일은 저장하지 않고, 추출 텍스트와 프로필, 분석 결과 JSON만 로컬 SQLite에 저장합니다.

다만 실제 출시 단계에서는 계정별 데이터 격리, 암호화, 개인정보 처리방침, 삭제 요청 처리, 로그 관리, 업로드 파일 검증이 반드시 필요합니다. 발표에서는 현재 로컬 데모의 처리 방식과 실제 출시 때 필요한 보완책을 분리해서 설명해야 합니다.

## 9. 이 프로젝트가 정보시스템 프로젝트라고 볼 근거는 무엇인가요?

DocMate는 단순 화면이 아니라 정보의 입력, 추출, 구조화, 판정, 저장, 재조회, 비교까지 이어지는 시스템입니다.

사용자 프로필과 공고 텍스트를 받아 정형 데이터로 바꾸고, 자격 조건과 비교한 뒤, 체크리스트와 원문 근거를 생성하고, SQLite 히스토리에 저장합니다. 즉, 공고 문서를 신청 준비 의사결정 데이터로 변환하는 정보 처리 파이프라인이 있습니다.

## 10. 왜 FastAPI나 React를 쓰지 않았나요?

현재 목표가 공개 상용 서비스가 아니라 경진대회 로컬 데모였기 때문입니다. 발표 데모에서는 설치와 실행이 안정적이어야 하므로 Python 표준 라이브러리 서버, no-build frontend, SQLite로 의존성을 줄였습니다.

공개 베타 단계에서는 FastAPI, React/TypeScript, PostgreSQL, 인증, 배포, CI/CD로 확장하는 것이 자연스럽습니다. 현재 스택은 부족해서가 아니라 데모 재현성을 위해 의도적으로 단순화한 선택입니다.

## 11. 사업성은 어디에 있나요?

개인 사용자의 일회성 결제 가능성은 낮다고 봅니다. 초기 사업성은 대학 행정팀, 청년센터, 지자체 정책 안내 조직처럼 반복 상담과 공고 안내 업무를 줄이고 싶은 기관 파일럿에서 더 현실적입니다.

개인 사용자에게는 무료 또는 저가 도구로 접근성을 확보하고, 기관에는 공고 템플릿 관리, 상담 보조, 관리자 검수, 통계 기능을 제공하는 방향이 더 설득력 있습니다.

## 12. 실제 출시까지 가장 큰 리스크는 무엇인가요?

가장 큰 리스크는 실공고 형식의 다양성, OCR이 필요한 PDF, 개인정보 처리, 계정 격리, 운영 안정성입니다.

현재 버전은 경진대회 발표용 로컬 MVP입니다. 실제 출시를 위해서는 실공고 평가셋, OCR, 클라우드 배포, 인증, 데이터 보안, 모니터링, 백업, 기관별 예외 규칙 관리가 필요합니다.

## 13. LLM을 붙이면 차별점이 사라지지 않나요?

아닙니다. LLM은 DocMate의 제품 가치를 대체하는 것이 아니라 보조할 수 있습니다.

DocMate의 차별점은 LLM 사용 여부가 아니라 결과 스키마, 신청 준비 흐름, 원문 근거, 체크리스트, 히스토리 비교, 보수적 판정 구조입니다. 향후 LLM은 라벨이 불분명한 문장이나 복잡한 예외 조항 해석에 제한적으로 사용하고, 최종 결과는 DocMate의 고정 스키마와 근거 검증 흐름을 통과시키는 방식이 적합합니다.

## 14. 발표에서 어떤 순서로 시연하는 것이 좋나요?

기능 목록을 나열하기보다 사용자 판단 흐름으로 보여주는 것이 좋습니다.

1. 프로필 작성: 사용자가 누구인지 정의한다.
2. 결과 요약: 신청 가능성과 위험 조건을 먼저 본다.
3. 원문 근거: 자동 분석 결과를 확인한다.
4. 체크리스트: 준비해야 할 행동으로 바꾼다.
5. 히스토리 비교: 여러 공고 중 어떤 것을 우선할지 비교한다.

이 흐름은 문제 정의, 분석 가치, 신뢰성, 실행성, 확장성을 한 번에 보여준다.

## 15. 한 문장으로 DocMate를 설명하면?

> DocMate는 장학금·청년정책 공고를 읽어야 하는 문서에서 바로 준비할 수 있는 신청 체크리스트와 판단 근거로 바꾸는 로컬 분석 콘솔입니다.

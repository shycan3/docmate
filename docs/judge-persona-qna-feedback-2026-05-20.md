# 5인 심사위원 페르소나 문답 및 피드백 반영 기록

작성일: 2026-05-20  
목적: 발표 심사에서 나올 수 있는 핵심 질문을 사전에 압박 테스트하고, 답변 논리와 문서 표현을 개선한다.

## 진행 방식

이번 문답은 실제 심사 상황을 가정한 페르소나 기반 Q&A 시뮬레이션이다. 각 심사위원은 서로 다른 평가 관점을 갖도록 설정했다.

- 기술 구현과 아키텍처
- AI/문서분석 방법론
- 정책/공공서비스 적합성
- UX/서비스 완성도
- 사업성/개인정보/출시 가능성

문답 후 합리적인 피드백은 관련 문서에 반영했다. 이번 라운드의 수정은 주로 코드 기능 추가가 아니라 발표 리스크를 줄이는 설명 보강과 문서 정리에 집중했다.

## J1. 김도현 심사위원: 시스템 구조와 재현성

### Q1. 이 프로젝트가 단순 웹페이지가 아니라 정보시스템 프로젝트라고 볼 근거는 무엇인가요?

답변: DocMate는 단순 화면이 아니라 입력 수집, 문서 파싱, 구조화 추출, 프로필 기반 판정, 체크리스트 생성, SQLite 히스토리 저장, 공고 비교까지 이어지는 정보 처리 흐름을 갖는다. 사용자의 신청 준비 의사결정을 돕기 위해 비정형 공고를 정형 데이터와 실행 항목으로 변환한다는 점에서 정보시스템 성격이 있다.

피드백: "정보시스템"이라는 표현을 쓰려면 데이터 흐름과 저장 구조를 분명히 보여줘야 한다. 발표에서 기능 목록만 말하면 웹 도구처럼 보인다.

반영: `docs/analysis-methods-and-tech.md`의 전체 흐름도와 데이터 모델 흐름을 발표 근거로 사용한다. 중요한 Q&A 문서에 "정보시스템으로서의 근거" 답변을 추가했다.

### Q2. 왜 FastAPI나 React 같은 일반적인 스택을 쓰지 않았나요?

답변: 현재 목표는 공개 서비스가 아니라 경진대회 로컬 데모다. 그래서 외부 설정과 빌드 실패 가능성을 줄이고, Python 표준 라이브러리 서버와 no-build frontend로 재현성을 우선했다. 공개 베타 단계에서는 FastAPI, React/TypeScript, PostgreSQL로 확장하는 로드맵을 갖고 있다.

피드백: 스택을 "부족해서 못 쓴 것"이 아니라 "데모 목적에 맞춰 의도적으로 단순화했다"라고 말해야 한다.

반영: 중요한 Q&A 문서에 이 답변을 포함했다.

### Q3. 저장되는 데이터는 무엇이고, 원본 PDF는 저장하나요?

답변: 원본 파일 자체는 저장하지 않는다. 현재 히스토리에는 추출 텍스트, 프로필, 추출 결과, 판정 결과, 위험 조건, 체크리스트, 원문 근거, 액션 링크가 JSON 형태로 SQLite에 저장된다.

피드백: 개인정보와 보안 질문으로 바로 이어질 수 있으니 "원본 파일 미저장"과 "출시 단계에서는 암호화/계정 격리 필요"를 같이 말해야 한다.

반영: 중요한 Q&A 문서에 개인정보 답변을 보강했다.

## J2. 박서연 심사위원: AI/NLP와 문서 분석

### Q1. 생성형 AI에 PDF를 넣고 분석해 달라고 하는 것과 무엇이 다른가요?

답변: 생성형 AI는 범용 대화형 분석 도구이고, DocMate는 장학금·청년정책 신청 준비에 특화된 워크플로우 도구다. DocMate는 매번 다른 답변을 생성하는 대신, 신청 상태, 위험 조건, 준비 서류, 원문 근거, 체크리스트, 히스토리 비교를 항상 같은 형식으로 제공한다.

피드백: "ChatGPT보다 낫다"라고 말하면 방어가 어렵다. "범용 요약이 아니라 도메인 업무 흐름을 제품화했다"가 더 안전하고 설득력 있다.

반영: `docs/differentiation-vs-generative-ai.md`에 피해야 할 표현과 사용할 표현을 추가했다.

### Q2. 모든 PDF를 분석할 수 있나요?

답변: 아니다. 현재는 텍스트 추출이 가능하고, 신청 기간·지원 대상·제출 서류 같은 구조가 어느 정도 있는 공고에 적합하다. 스캔 이미지 PDF, 표/각주 중심 문서, 자유서술형 보고서는 한계가 크다.

피드백: 이 한계는 숨기면 안 된다. 오히려 "지원 범위"를 먼저 말해야 신뢰가 생긴다.

반영: `docs/analysis-methods-and-tech.md`에 `지원 범위와 비지원 범위` 섹션을 추가했다.

### Q3. 규칙 기반이면 정확도 검증은 어떻게 하나요?

답변: 현재는 샘플 공고와 테스트 중심이다. 실제 서비스 신뢰도는 실공고 fixture를 모아 항목 추출 정확도, 자격 판정 안전성, 위험 조건 탐지, 원문 근거 품질을 측정해야 한다.

피드백: 발표에서 정확도 수치를 말할 수 없다면, 최소한 측정 계획은 있어야 한다.

반영: `docs/analysis-methods-and-tech.md`에 `검증 계획` 표를 추가했다.

## J3. 이민정 심사위원: 정책 실무와 사용자 위험

### Q1. 신청 가능이라고 나왔는데 실제로는 불가능하면 책임 문제가 생기지 않나요?

답변: 그래서 DocMate는 애매한 조건을 가능하다고 단정하지 않고 `추가 확인 필요`로 둔다. 결과에는 원문 근거와 확인 안내를 붙여 최종 판단은 원문 공고와 기관 안내를 확인하도록 설계했다.

피드백: "신청 가능"이라는 문구는 강하게 들린다. 발표에서는 "기본 조건 기준 신청 가능"이라고 설명하고, 최종 권한은 기관에 있음을 말해야 한다.

반영: 중요한 Q&A 문서에 보수적 표현을 포함했다. 향후 UI 문구도 `기본 조건 충족`처럼 조정할 후보로 기록했다.

### Q2. 청년정책 공고는 자주 바뀌는데, 최신성은 어떻게 보장하나요?

답변: 현재 데모는 사용자가 업로드한 문서 또는 내장 샘플을 분석한다. 따라서 최신성은 사용자가 넣은 공고 파일에 달려 있다. 실제 서비스 단계에서는 공고 수집 파이프라인, 갱신일 표시, 기관 원문 링크 검증이 필요하다.

피드백: "자동으로 최신 정책을 가져온다"는 인상을 주면 안 된다.

반영: 중요한 Q&A 문서에 최신성 답변을 추가했다.

### Q3. 실제 정책 상담 현장에서 쓸 수 있나요?

답변: 현재는 상담원을 대체하기보다는 1차 정리 도구에 가깝다. 상담 전 사용자가 공고의 핵심 조건과 준비 서류를 정리해 오도록 도와 반복 질문을 줄일 수 있다. 기관용으로 확장하려면 검수 워크플로우와 관리자 대시보드가 필요하다.

피드백: B2B 가능성을 말하되, 현재 기능과 미래 기능을 분리해야 한다.

반영: 중요한 Q&A 문서와 사업성 답변에 "현재/미래" 경계를 명시했다.

## J4. 최윤아 심사위원: UX와 발표 시연

### Q1. 사용자가 프롬프트를 쓰지 않아도 된다는 점은 좋은데, 첫 화면에서 그 가치가 바로 보이나요?

답변: 최근 UI는 프로필 작성과 공고 선택을 2단계로 단순화했고, 직업/거주지/소득구간은 선택형 입력으로 바꿨다. 사용자가 "무엇을 넣고 무엇을 받는지"를 한 화면에서 알 수 있게 했다.

피드백: 발표에서는 첫 화면을 길게 설명하지 말고, 바로 프로필 선택과 샘플 분석으로 넘어가야 한다.

반영: 중요한 Q&A 문서에 "발표 시연 우선순위" 답변을 추가했다.

### Q2. 결과 화면 정보가 많아 보일 때 어떤 순서로 설명해야 하나요?

답변: 결과 요약 → 원문 근거 → 체크리스트 → 히스토리 비교 순서가 좋다. 이는 사용자의 의사결정, 신뢰 확인, 다음 행동, 여러 공고 비교를 차례로 보여준다.

피드백: 발표 흐름은 기능 나열이 아니라 사용자 판단 흐름이어야 한다.

반영: 중요한 Q&A 문서에 발표 4단계 흐름을 추가했다.

### Q3. 체크리스트가 실제 사용자의 행동으로 이어지는 근거가 있나요?

답변: 현재는 기능적 가설이다. 제출 서류와 마감일 확인을 별도 목록으로 바꾸면 사용자가 놓치는 일을 줄일 수 있다는 가정이며, 실제 효과는 사용자 연구에서 측정해야 한다.

피드백: "효과가 있다"가 아니라 "측정할 수 있는 가설"로 말하면 더 신뢰가 간다.

반영: 검증 계획에 사용자 효용 측정을 포함했다.

## J5. 정우석 심사위원: 사업성, 개인정보, 출시 가능성

### Q1. 개인 사용자가 돈을 낼까요?

답변: 개인의 일회성 결제 가능성은 낮다고 본다. 초기 사업성은 B2C보다 대학 행정팀, 청년센터, 지자체 정책 안내 조직처럼 반복 상담을 줄이고 싶은 기관 파일럿에서 더 현실적이다.

피드백: 개인 구독보다 기관 파일럿을 먼저 말하는 편이 설득력 있다.

반영: 중요한 Q&A 문서에 사업성 답변을 보강했다.

### Q2. 실제 출시까지 가장 큰 리스크는 무엇인가요?

답변: 실공고 파싱 정확도, OCR, 개인정보 처리, 계정 격리, 배포/운영 안정성이다. 현재 데모는 발표 가능한 로컬 MVP이고, production-ready 서비스는 아니다.

피드백: 발표에서 "출시 가능"을 말할 때는 경진대회 로컬 데모와 실제 상용 출시를 구분해야 한다.

반영: 중요한 Q&A 문서에 출시 가능성 답변을 추가했다.

### Q3. LLM을 붙이면 오히려 차별점이 사라지지 않나요?

답변: LLM을 붙여도 DocMate의 차별점은 유지된다. LLM은 비정형 문장 해석을 보조하고, DocMate는 결과 스키마, 원문 근거, 체크리스트, 히스토리 비교, 보수적 판정 흐름을 유지한다. 즉 LLM은 엔진 일부이고, 제품 가치는 신청 준비 워크플로우다.

피드백: 향후 하이브리드 전략을 "LLM으로 대체"가 아니라 "LLM을 제한적으로 보조"한다고 표현해야 한다.

반영: `docs/differentiation-vs-generative-ai.md`의 하이브리드 전략과 중요한 Q&A 문서에 반영했다.

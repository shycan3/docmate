# 발표 스크린샷 사용 가이드

작성일: 2026-05-21  
대상 폴더: `presentation-materials/screenshots/`  
캡처 기준: 로컬 발표 데모 버전

이 문서는 기존 `screenshots/` 폴더에 있는 발표용 스크린샷 8장의 역할을 정리한 자료입니다. `images/` 폴더가 실제 서비스 화면을 자연스럽게 보여주는 기본 캡처라면, `screenshots/` 폴더는 발표자가 특정 기능과 심사 포인트를 빠르게 설명할 수 있도록 장면을 선명하게 잡은 자료입니다.

## 추천 발표 흐름

1. `01-prep-dashboard.png`로 "무엇을 하는 서비스인지"를 먼저 보여준다.
2. `02-demo-progress.png`를 짧게 보여주며 실제 분석이 진행되는 느낌을 만든다.
3. `05-evidence-matching.png`로 신뢰성 질문에 선제 대응한다.
4. `04-risk-detection.png`로 사용자가 놓치기 쉬운 제한 조건을 강조한다.
5. `06-checklist-export.png`로 분석이 실제 신청 준비 행동으로 이어진다는 점을 말한다.
6. `03-compare-dashboard.png`로 단일 공고 분석을 넘어 여러 기회를 비교하는 확장성을 보여준다.
7. 시간이 남거나 기술 질문이 들어오면 `07-automation-terminal.png`, `08-sqlite-schema.png`를 사용한다.

## 슬라이드 배치 팁

- `01`, `03`, `05`, `06`은 화면 자체가 서비스 가치 설명에 직접 연결되므로 크게 배치한다.
- `04-risk-detection.png`는 이미 콜아웃이 포함된 자료이므로 추가 도형을 많이 얹지 않는 편이 좋다.
- `07-automation-terminal.png`와 `08-sqlite-schema.png`는 발표 본문보다 Q&A 또는 부록 슬라이드에 적합하다.
- 실제 시연이 끊겼을 때를 대비해 `02-demo-progress.png`와 `03-compare-dashboard.png`를 백업 이미지로 준비하면 안전하다.
- 심사위원이 "정확도와 근거는 어떻게 보장하나?"라고 물으면 `05-evidence-matching.png`를 먼저 보여주고, 이어서 `04-risk-detection.png`를 보여준다.

## 이미지 세트별 차이

| 폴더 | 쓰임 | 특징 |
| --- | --- | --- |
| `images/` | 자연스러운 실제 서비스 화면 소개 | 사용자가 보는 제품 화면을 그대로 설명하기 좋다. |
| `screenshots/` | 발표·심사 대응용 핵심 장면 | 기능별로 보여줄 지점이 선명하고, 콜아웃이나 구현 근거를 포함한다. |
| `visuals/` | 개념도와 비교도 | 서비스 구조, 분석 파이프라인, 생성형 AI와의 차별점처럼 추상적 내용을 설명한다. |

## 발표 멘트 예시

`05-evidence-matching.png`를 보여줄 때:

> DocMate는 결과만 보여주는 방식이 아니라, 왜 그렇게 판단했는지 원문 근거를 함께 보여줍니다. 자동화 결과를 그대로 믿으라는 것이 아니라, 사용자가 근거를 확인하고 신청 여부를 판단할 수 있게 만드는 구조입니다.

`04-risk-detection.png`를 보여줄 때:

> 신청 실패는 혜택을 몰라서가 아니라, 제한 조건이나 제출 서류를 놓쳐서 생기는 경우가 많습니다. 그래서 DocMate는 가능 여부만 보여주지 않고, 중복지원 제한이나 추가 확인이 필요한 조건을 별도로 분리합니다.

`06-checklist-export.png`를 보여줄 때:

> 분석의 마지막 결과는 긴 요약문이 아니라 할 일 목록입니다. 사용자는 제출 서류와 다음 행동을 바로 확인하고 신청 준비를 시작할 수 있습니다.

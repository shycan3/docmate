# 예상 심사 질문과 답변 방향

## Q1. 단순 요약 서비스와 무엇이 다른가요?

DocMate는 요약문 생성이 아니라 신청 의사결정과 실행 준비에 초점을 둡니다. 신청 가능 여부, 위험 조건, 제출 서류, 체크리스트, 원문 근거, 히스토리 비교까지 하나의 흐름으로 제공합니다.

## Q2. 판정 결과를 신뢰할 수 있나요?

현재는 발표용 로컬 데모이므로 규칙 기반 판정과 샘플 공고 중심입니다. 대신 결과마다 원문 근거를 표시해 사용자가 판정 이유를 확인할 수 있게 했습니다. 실제 서비스 단계에서는 실공고 데이터셋, 회귀 테스트, 기관별 예외 규칙, 전문가 검수 흐름이 필요합니다.

## Q3. 개인정보는 어떻게 다루나요?

현재 데모는 로컬 환경에서 실행되고, 원본 파일은 저장하지 않으며 추출 텍스트·프로필·분석 결과만 로컬 SQLite에 저장합니다. 출시 단계에서는 개인정보 처리방침, 데이터 최소수집, 암호화, 계정별 격리, 삭제 요청 처리, 로그 관리가 필요합니다.

## Q4. 실제 사용자가 돈을 낼 만한 서비스인가요?

개인 사용자는 무료 또는 저가 구독보다 접근성이 중요합니다. 더 현실적인 초기 시장은 대학 행정팀, 청년센터, 지자체 정책 안내 조직처럼 반복 상담과 공고 안내 업무를 줄이고 싶은 기관입니다.

## Q5. 공고 형식이 제각각이면 어떻게 대응하나요?

현재는 텍스트 기반 PDF/TXT와 샘플 데이터를 우선 처리합니다. 향후에는 OCR, 기관별 템플릿, 항목 추출 품질 평가, 예외 규칙 관리가 필요합니다. 발표에서는 이 한계를 명확히 말하되, 현재 구조가 추출·판정·근거·체크리스트로 분리되어 확장 가능하다는 점을 강조합니다.

## Q6. 왜 AI 모델을 바로 붙이지 않았나요?

경진대회 데모에서는 안정적으로 재현 가능한 결과가 중요하다고 판단했습니다. 그래서 규칙 기반 파이프라인을 먼저 만들고, 어디서 어떤 결과가 나오는지 설명 가능하게 구성했습니다. 이후 LLM은 애매한 조건 해석, 자연어 질의응답, 기관별 문체 대응에 보조적으로 붙이는 방향이 적합합니다.

## Q7. 발표 중 꼭 보여줘야 할 장면은 무엇인가요?

`프로필 작성`, `결과 요약`, `원문 근거`, `체크리스트`, `히스토리 비교`입니다. 이 다섯 장면이 문제 정의부터 실행 가치까지 가장 짧게 전달합니다.

## Q8. 현재 가장 큰 한계는 무엇인가요?

실제 출시 기준으로는 배포, 계정, 개인정보, 대량 실공고 검증, OCR, 운영 모니터링이 부족합니다. 다만 경진대회 발표 기준으로는 로컬에서 서비스 흐름이 구동되고, 피드백을 반영한 데모 경험이 준비되어 있습니다.
